Java中的IO体系架构与装饰模式

发表于 2018-04-29 | 分类于 Java ， 设计模式 | 阅读次数: | 本文总阅读量 次

输入/输出流的概念

输入/输出时，数据在通信通道中流动。所谓“数据流（stream）”指的是所有数据通信通道中，数据的起点和终点。信息的通道就是一个数据流。只要是数据从一个地方“流”到另一个地方，这种数据的通道都可以成为数据流。输入/输出是相对于程序来说的。程序在使用数据时所扮演的角色有两个：一个是源，一个是目的。若程序是数据流的源，即数据流的提供者，这个数据流对程序来说就是一个“输出数据流”（数据从程序流出）。若程序是数据流的终点，这个数据流对程序而言就是一个“输入数据流”（数据从程序外流向程序）。

流的分类

  1. 从功能上分：输入流与输出流
  2. 从结构上分：
  • 字节流 ：以字节为处理单位或称面向字节。其输入和输出的基础是InputStream和OutputStream这两个抽象类。字节流的输入输出操作由这两个类的子类实现。
  • 字符流 ：以字符为处理单位或称面向字符。其输入和输出的基础是Reader和Writer这两个抽象类。字符流的输入和输出操作由这两个类的子类实现。
  1. 其他维度分：
  • 节点流：从特定地方读写的流类，如磁盘或一块内存区域
  • 过滤流：使用节点流作为输入输出。过滤流是使用一个已经存在的输入流或者输出流连接创建的。
  • java.io包中的输入流区分节点流与过滤流
  • java.io包中的输出流区分子节点与过滤流

Java I/O库的设计原则

Java的I/O库给我们提供了一个称作连接的机制。可以将一个流与另一个流首尾相接，形成一个流管道的链接。这种机制实际上是一种被称为Decorator（装饰）设计模式的应用。
装饰模式又名包装（Wrapper）模式。它以对客户端透明的方式扩展对象的功能，动态的给一个对象附加上更多的责任，换言之，客户端并不会觉得对象在装饰前和装饰后有什么不同。它是继承关系的一个替代方案，可以在不创建更多子类的情况下，通过创建一个包装对象来将对象的功能动态的加以扩展。
装饰模式把客户端的调用委派到被装饰类。装饰模式的关键在于这种扩展是完全透明的。

装饰模式的角色

  • 抽象构件角色（Component）：给出一个抽象接口，以规范准备接收附加责任的对象。
  • 具体构件角色（Concrete Component）：定义一个将要接收附加责任的类。
  • 装饰角色（Decorator）：持有一个构件（Component）对象的引用，并定义一个与抽象构件接口一致的接口。
  • 具体装饰角色（Concrete Decorator）：负责给构件对象“贴上”附加的责任。

代码实现：
抽象构件角色：

public interface Component {
    void doSomeThing();
}

具体构件角色:

public class ConcreteComponent implements Component {
    @Override
    public void doSomeThing(){
        System.out.println("Function A:具体构件实现的功能");
    }
}

装饰角色:

public class Decorator implements Component {
    //持有一个构件对象的引用
    private Component component;

    public Decorator (Component component) {
        this.component = component;
    }
    //通过实现构件对象接口，来定义一个与抽象构件接口一致的接口
    @Override
    public void doSomeThing() {
        component.doSomeThing();
    }
}

具体装饰角色B:

public class ConcreteDecoratorB extends Decorator {
    @Override
    public void doSomeThing() {
        super.doSomeThing();
        this.doAnotherThing();
    }

    public void doAnotherThing() {
        System.out.println("Function B：具体装饰角色包装的功能");
    }
}

具体装饰角色C:

public class ConcreteDecoratorC extends Decorator {
    @Override
    public void doSomeThing() {
        super.doSomeThing();
        this.doAnotherThing();
    }

    public void doAnotherThing() {
        System.out.println("Function C: 具体装饰角色包装的功能");
    }
}

客户端：

public class Client {
    public static void main (String[] args) {
        Component component = new CocreteDecoratorB(new ConcreteDecoratorC(new ConcreteComponent()));
        component.doSomeThing();
    }
}

装饰模式的特点

  • 装饰对象和真是对象有相同的接口，如示例中的Decorator对象有与Component对象一样的doSomeThing接口。这样客户端就可以已和真实对象相同的方式和装饰对象交互。
  • 装饰对象包含一个真是对象的引用，如示例中Decorator持有一个Component类型的字段（private Component component;）
  • 装饰对象接收所有来自客户端的请求，把这些请求转发给真实的对象，示例中通过重写抽象构件的doSomeThing方法，调用super.doSomeThing()方法实现。
  • 装饰对象可以在转发这些请求以前或以后增加一些附加功能。这样就确保了在运行时，不用修改给定对象的结构就可以在外部增加附加的功能。如实例中在重写的doSomeThing方法中调用了具体装饰角色的doAnotherThing()方法。在面向对象设计中，通常是通过继承来实现对给定类的功能扩展。

装饰模式与继承的区别

装饰模式：

  • 用来扩展对象的功能
  • 不需要子类
  • 动态的
  • 运行时分配职责
  • 防止由于子类而导致的复杂和混乱
  • 更多的灵活性
  • 对于一个给定的对象，同时可能有多个不同的装饰对象，客户端可以通过它的需要选择合适的装饰对象发送消息

继承：

  • 用来扩展一类对象的功能
  • 需要子类
  • 静态的
  • 编译时分配职责
  • 导致很多子类产生
  • 缺乏灵活性

装饰模式的适用情况

  • 想要透明且动态的给对象增加新的职责（方法）而又不影响其他对象。
  • 给对象增加的职责在未来可能会发生变化。
  • 用子类扩展功能不实际的情况下。
